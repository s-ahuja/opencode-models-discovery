# v0.12.0: Provider-Level Discovery Migration Plan

## Goal

Version `0.12.0` is a transition release before the larger `1.0.0` change.

The `1.0.0` target is to remove plugin-level global discovery configuration. Discovery should remain enabled by default, and users can customize or disable behavior per provider through `provider.<name>.options.modelsDiscovery`.

The `0.12.0` release should keep existing behavior compatible while warning users about the upcoming breaking change and providing a migration command that can move global discovery settings onto concrete provider configuration.

## Non-Goals

- Do not remove global plugin configuration in `0.12.0`.
- Do not change discovery defaults in `0.12.0`.
- Do not implement the `1.0.0` environment-variable default switch in `0.12.0` unless the release scope changes.
- Do not guess or rewrite providers that do not come from editable config.
- Do not silently overwrite existing provider-level `modelsDiscovery` settings.

## Legacy Global Configuration

The following plugin-level options are considered legacy global discovery configuration:

- `discovery.enabled`
- `providers.include`
- `providers.exclude`
- `models.includeRegex`
- `models.excludeRegex`
- `smartModelName`

These options currently affect discovery across providers. In `1.0.0`, discovery behavior should live on the provider itself.

## 1.0.0 Target Defaults

After plugin-level global discovery configuration is removed in `1.0.0`, discovery should still be enabled by default for providers unless explicitly disabled.

Recommended provider-level precedence:

1. `provider.<id>.options.modelsDiscovery.enabled` when explicitly set.
2. `OPENCODE_MODELS_DISCOVERY_DEFAULT_ENABLED` when set.
3. Plugin built-in default: `true`.

This means:

- `modelsDiscovery.enabled: true` explicitly enables discovery for that provider.
- `modelsDiscovery.enabled: false` explicitly disables discovery for that provider.
- Missing `modelsDiscovery.enabled` uses the environment-variable default if present, otherwise discovery remains enabled.

The environment variable should control only the default for providers without explicit provider-level configuration. It should not override `modelsDiscovery.enabled: true` or `modelsDiscovery.enabled: false`.

Suggested environment variable:

```sh
OPENCODE_MODELS_DISCOVERY_DEFAULT_ENABLED=false
```

Suggested accepted false values: `false`, `0`, `no`, `off`.
Suggested accepted true values: `true`, `1`, `yes`, `on`.
Invalid values should warn and fall back to the built-in default `true`.

This keeps the plugin easy to use by default while still letting advanced users or automation environments make discovery opt-in globally without reintroducing plugin-level JSON configuration.

## Transition Behavior

When the plugin receives legacy global discovery configuration, it should:

- Preserve current behavior.
- Log a warning.
- Show a toast warning once per plugin instance after the host is ready for UI events.
- Inject a migration command into runtime config.

Suggested toast message:

```text
Global opencode-models-discovery config will be deprecated in v1.0.0. Use provider.<name>.options.modelsDiscovery instead. Run /models-discovery:migrate to migrate config.
```

The toast should not use a fixed delay from the config hook. Fixed delays can be too early when startup has more work to do and too late when the UI is ready quickly.

Initial testing showed that `server.connected` is not consistently delivered to server plugins in the current runtime. Observed startup events included `plugin.added`, `connector.updated`, and `reference.updated`. The implementation should show the pending toast on `reference.updated` and may also accept `server.connected`, `workspace.ready`, and `worktree.ready` for compatibility with other runtimes and newer event streams.

## Migration Command

The plugin should inject a runtime command when legacy global config is present:

```json
{
  "command": {
    "models-discovery:migrate": {
      "description": "Migrate opencode-models-discovery config",
      "agent": "build",
      "template": "Use the customize-opencode skill. Migrate opencode-models-discovery legacy global configuration to provider-level configuration..."
    }
  }
}
```

OpenCode commands are prompt templates, not direct TypeScript callbacks. The command should guide the assistant to use the built-in `customize-opencode` skill and migrate the actual editable OpenCode config files with normal agent tools and permissions.

The command should:

- Inspect project config files: `opencode.json`, `opencode.jsonc`, and `.opencode/opencode.json` under the current project/worktree.
- Inspect the user global config: `~/.config/opencode/opencode.json`.
- Inspect `OPENCODE_CONFIG` if it is set and points to a file.
- Avoid editing managed or organization-controlled config unless the user explicitly asks for it.
- Preserve unrelated config and formatting as much as possible.
- Refuse to guess provider ids that are not present in editable config.
- Preserve existing provider-level `modelsDiscovery` fields unless the user explicitly requests overwrite.
- Remind the user to restart opencode after config edits.

Managed config locations are platform-specific and should be mentioned as non-default edit targets:

- macOS: `/Library/Application Support/opencode/`
- Linux: `/etc/opencode/`
- Windows: `%ProgramData%\opencode`

## Provider Source Discovery

The command-driven migration should prefer providers declared in editable config files. It should not rely solely on resolved merged config because provider source and plugin options can come from different layers.

If a future implementation reintroduces a direct migration tool, it should call `client.config.providers({ directory, workspace })` and inspect provider source metadata.

Providers with these sources should be skipped unless a later implementation can safely resolve an editable config source:

- `env`
- `custom`
- `api`

The resolved provider list helps avoid writing provider-level settings for providers that do not actually live in user-editable config.

## Config Source Complexity

OpenCode merges config from multiple layers. Later layers override earlier ones.

Relevant source order:

1. Remote config from `.well-known/opencode`
2. Global config at `~/.config/opencode/opencode.json`
3. Custom config from `OPENCODE_CONFIG`
4. Project config `opencode.json`
5. `.opencode` directories
6. Inline config from `OPENCODE_CONFIG_CONTENT`
7. Managed config files
8. macOS managed preferences

Because resolved config is merged, it may not reveal exactly which file contributed a specific provider or plugin option. Migration must locate editable config files deliberately.

## Config Write Strategy

For `target: "auto"`, use this order:

1. If a project config file contains the provider, write provider-level config there.
2. Else if the global config file contains the provider, write provider-level config there.
3. Else if `OPENCODE_CONFIG` points to a config file containing the provider, consider writing it only if the user selected `target: "project"` or a future explicit custom target.
4. Otherwise skip the provider and report that no editable provider source was found.

The command-driven approach can let the assistant handle JSON and JSONC via normal editing tools, but it should still be conservative when comments or formatting make a safe automated rewrite unclear.

Direct TypeScript migration tooling, if added later, should initially support JSON config files only:

- `opencode.json`
- `~/.config/opencode/opencode.json`

If a matching `.jsonc` file is found, skip automatic edits and report that JSONC migration is not supported yet.

## Field Mapping

Global fields map to provider-level fields as follows:

| Legacy global field | Provider-level field |
| --- | --- |
| `discovery.enabled` | `provider.<id>.options.modelsDiscovery.enabled` |
| `models.includeRegex` | `provider.<id>.options.modelsDiscovery.models.includeRegex` |
| `models.excludeRegex` | `provider.<id>.options.modelsDiscovery.models.excludeRegex` |
| `smartModelName` | `provider.<id>.options.modelsDiscovery.smartModelName` |

Provider filter handling:

- `providers.include`: included providers get `modelsDiscovery.enabled = true`.
- `providers.exclude`: excluded providers get `modelsDiscovery.enabled = false`.
- Non-included providers should not be forced to `enabled = false` by default.

## Conflict Rules

Existing provider-level settings must be preserved by default.

If a provider already has `options.modelsDiscovery` values:

- Keep existing values.
- Explain skipped fields in the migration response.
- Only replace them when the user explicitly requests overwrite.

Removing legacy plugin config should be part of the migration only when the assistant has safely moved the relevant fields or clearly explains the manual change. `0.12.0` remains a transition release and must preserve runtime compatibility.

## Implementation Steps

1. Add legacy global config detection.
2. Add one-time toast warning state in the config hook and show it from a host-ready event.
3. Inject `models-discovery:migrate` command while preserving existing commands.
4. Have the command launch `customize-opencode` guidance for project/global/custom config migration.
5. Add tests for detection, command injection, toast warning, and command conflict behavior.
6. Update README and configuration docs with the `0.12.0` transition notice.

## Open Questions

- Should the command recommend `OPENCODE_MODELS_DISCOVERY_DEFAULT_ENABLED=false` whenever `providers.include` is used, or only when a pure config migration would require many disable entries?
- Should future releases add a direct TypeScript migration tool after observing command-driven migration behavior?
